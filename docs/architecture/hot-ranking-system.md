## HOT 이벤트 & 아티스트 시스템

### **문제 상황**

- 메인 페이지에서 **인기 이벤트 / 아티스트**를 노출하기 위해

  조회수·북마크·참여 지표를 종합한 **실시간 랭킹 산정**이 필요했습니다.

- 초기 구현에서는
  - 매 요청마다 DB에서 집계·정렬을 수행하며
  - 트래픽 증가 시 **정렬 쿼리 비용과 DB 부하가 급격히 증가**하는 문제가 있었습니다.
- 특히 메인 페이지 특성상 **읽기 빈도는 높고, 순위 변동은 잦은** 구조로

  단순 캐싱이나 DB 정렬만으로는 한계가 있었습니다.


---

### **해결 방법**

### **1) Redis ZSET 기반 랭킹 집계 구조 설계**

- 이벤트 / 아티스트별 점수를 **Redis ZSET(score, member)** 구조로 관리
- 점수는 다음 지표를 조합해 산정
  - 조회수
  - 북마크 수
  - 사용자 참여 지표
- 점수 변경 시 ZSET에 **증분 반영**하여
  - 전체 재계산 없이 순위 갱신 가능하도록 설계

→ DB 정렬 없이도 **O(log N)** 복잡도로 실시간 순위 변동 처리 가능

---

### **2) “선정”과 “조회”의 책임 분리**

- Hot 랭킹 시스템의 역할은
  - **노출 대상 ID 집합을 산출하는 것까지로 한정**
- 메인 페이지에서는
  - Redis ZSET에서 **상위 ID 목록만 조회**
  - 실제 엔티티 조회 및 로딩 방식은

    **메인 페이지 API 성능 개선 구조(ID 기반 2단계 조회)에 위임**


→ 랭킹 로직과 조회 최적화를 분리해

각 계층을 **독립적으로 최적화 가능**하도록 구성

---

### **3) 추천 신뢰도 확보를 위한 제어 장치**

- 일시적 트래픽 급증이나 단일 지표 편향을 방지하기 위해
  - 점수 가중치 조정
  - 최소 노출 기준 설정
- 이를 통해
  - 단순 “많이 본 콘텐츠”가 아닌
  - **의미 있는 사용자 반응을 반영한 추천 결과**를 유지

---

### **성과**

- 메인 페이지 랭킹 조회 시
  - DB 정렬 쿼리 제거
  - 랭킹 산정 비용을 Redis로 완전히 분리
- 트래픽 증가 시에도
  - 랭킹 계산 비용이 DB에 영향을 주지 않는 구조 확보
- 랭킹 산정 로직과 조회 로직을 분리해
  - **메인 페이지 성능 개선 작업과 독립적으로 진화 가능한 구조** 완성
